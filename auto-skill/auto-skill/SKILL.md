# Auto-Skill：自动化 Skill 构建与优化

## 最高优先级执行卡

只要本 Skill 被触发，先执行本节；后文的理念、模板、脚本说明都不能覆盖本节。

### 何时必须调用

用户出现以下任何意图时，必须使用 Auto-Skill：
- 创建一个新 Skill
- 优化、打磨、进化、校准已有 Skill
- 评估一个 Skill 是否有效
- 给 Skill 做 benchmark、真实案例测试、回归测试
- 根据测试结果反推 Skill 设计缺陷
- 比较 v0/v1/v2 等不同版本效果

### 绝对禁止

- 禁止只凭静态阅读下结论
- 禁止没有真实案例就修改 Skill
- 禁止把 LLM 裁判建议当成用户裁决
- 禁止一轮测试后说“最终完成”，除非用户明确接受并选择停止
- 禁止在用户裁决前覆盖原 Skill
- 禁止只展示总结，不展示旧版和候选版的完整输出对比
- 禁止裁判只给主观结论；裁判必须按评估表逐项给证据、置信度和风险
- 禁止编辑者只凭裁判意见动手；编辑者输入必须综合“裁判建议 + 用户裁决/修正意见”
- 用户叫停、暂停或选择停止时，循环立即截止
- 禁止把平台专有能力写成 Skill 的通用依赖

### 角色分工

- **执行者 Runner**：只负责跑测试、保存完整输出。不能评价，不能改 Skill。
- **裁判 Judge**：只根据输出对比做判断，提出优化方向。不能替用户决定，不能直接改 Skill。
- **编辑者 Editor**：只在用户认可裁判方向后，基于“裁判建议 + 用户意见”的合成指令生成候选版。不能自行扩大修改范围。
- **用户 User**：唯一裁决者。每次裁判给出判断后，用户拍板；编辑者只能接收用户拍板后的合成意见。

### 必须先做的固定流程

1. 判断任务是“从零创建”还是“优化已有 Skill”
2. 建立评估契约：哪些指标机器可判，哪些由 LLM 辅助，哪些必须人工裁决
3. 设计真实测试用例：核心场景、边界场景、回归场景都要有
4. 执行者跑基线：无 Skill vs 有 Skill，或旧版 Skill vs 当前 Skill；保存完整输出
5. 裁判只看输出对比，判断 Skill 是否有效，并提出初步优化方向
6. **用户拍板**：看左右输出和裁判方向，决定是否交给编辑者；不认可则补测试、重判或停止
7. 监督者把“裁判建议 + 用户拍板意见”整理成编辑者输入摘要
8. 编辑者只按编辑者输入摘要生成候选版 candidate；候选版不覆盖原 Skill
9. 执行者跑旧版 vs 候选版的完整测试输出，裁判复核候选版是否执行了用户意见
10. **用户拍板**：决定接受修订、继续下一轮、回退、补测试或停止；用户叫停则循环截止

### 第一轮改动如何选

第一轮只改 1-3 个最有证据的问题，优先级如下：
1. 机器可明确判断的失败
2. 没有交付用户真正要的结果
3. 多个测试用例共同暴露的同一根因
4. 用户明确指出的问题
5. 体验和风格类小幅增强

### Gate 报告必须包含

- 测试用例原文
- 左边版本完整输出：without/current/original
- 右边版本完整输出：with/candidate/revised
- 逐项差异和副作用
- 哪些判断来自机器指标，哪些来自 LLM，哪些需要用户裁决
- 裁判评价表：评价维度、判断方法、证据、置信度、风险、副作用
- 裁判建议：左边好 / 右边好 / 都不够好 / 需要补测试
- 用户裁决区：接受方向 / 拒绝方向 / 接受修订并继续 / 接受修订并停止 / 回退 / 补测试

## 核心理念

Auto-Skill 是一个**元 Skill（Meta-Skill）**，它的目标不是直接解决用户的任务，
而是**自动化地构建和优化解决用户任务的 Skill**。它借鉴了 SkillOpt（Microsoft Research）
的"监督式 Skill 优化"思想，结合 skill-creator 的评估框架，形成一个完整的闭环：

```
自然语言描述
    ↓
Phase 0: 半自动 Interview → 生成 Skill 初版（v0）
    ↓
Phase 1: 基线测试 → 获得 with_skill vs without_skill 数据
    ↓
Phase 2: 优化循环（Runner → Judge → 用户拍板 → Editor → Runner → Judge → 用户拍板）
    ↓
用户裁决：接受并停止 / 继续下一轮 / 回退 / 补测试
```

## 三阶段架构

### Phase 0：Interview 与初版生成（半自动）

**目标**：用自然语言需求生成第一个可用的 Skill 初版。

**流程**：
1. 用户提供初始需求描述（如"我想做一个写作 Skill"）
2. 运行 Phase 0 Interview 脚本，提出结构化追问（见下方 Interview 模板）
3. 用户回答追问
4. 根据回答生成 Skill 初版 SKILL.md
5. **用户确认**：用户审阅初版内容，确认"是否准确捕获了我的意图"
6. 如不满意，进入修正轮次（用户指出问题 → 重新生成）
7. 确认后保存为 `outputs/{skill-name}/v0/SKILL.md`

**为什么半自动？**
- 用户需要看到初版才能判断是否准确
- 自动化 Interview 可能遗漏用户隐含的核心需求
- 用户确认是后续优化有效性的前提

### Phase 1：基线测试

**目标**：获得 Skill 性能的量化起点。

**流程**：
1. 先建立评估契约：区分机器可判指标、LLM 可辅助判断指标、必须人工裁决指标
2. 根据 Skill 类型设计测试用例（prompts）：快速基线 3-5 个，正式优化 8-12 个训练用例 + 3-5 个 held-out 验证用例
2. 对每个用例并行运行：
   - **with_skill**：使用当前版本的 Skill
   - **without_skill**：不使用任何 Skill（纯模型能力）
3. 收集输出、时间、token 消耗，以及可自动计算的通过/失败信号
4. 生成 benchmark 报告，明确哪些结论来自客观指标，哪些只是裁判建议
5. 保存到 `outputs/{skill-name}/baseline/`

## 评估契约：指标分层

Auto-Skill 是通用优化器，不默认所有任务都需要人工评价，也不默认 LLM 可以裁决所有质量问题。

每个 Skill 在进入基线测试前，必须先把评估指标分成三类：

### A 类：机器可明确判断

这类指标可以用脚本、测试、解析器或固定规则判断，优先级最高。

典型例子：
- 代码类：单元测试是否通过、lint 是否通过、类型检查是否通过、命令是否能运行
- 数据类：输出 JSON/CSV/SQL 是否可解析，字段是否齐全，数值是否在允许范围
- 文档/文件类：文件是否生成、是否能打开、页数/字数/标题层级是否符合要求
- 检索/问答类：是否引用指定来源、是否包含必需字段、是否没有超出允许引用长度
- 通用指标：耗时、token、错误率、重复率、格式通过率

要求：
- A 类指标必须写成明确的 pass/fail 或数值阈值
- 能跑 benchmark 就必须跑 benchmark，不能用主观描述替代
- A 类失败通常优先于 B/C 类优化，除非用户明确更重视主观体验

### B 类：LLM 可辅助判断，但不能单独裁决

这类指标可以由独立 LLM 裁判初评，但必须展示证据，不能只给结论。

典型例子：
- 是否遵循了 Skill 的步骤
- 结构是否完整
- 是否遗漏明显约束
- 是否存在明显自相矛盾
- 是否有明显 AI 套话、模板化、过度解释
- 分析类输出是否有明显逻辑跳跃

要求：
- LLM 裁判必须引用具体输出片段作为证据
- LLM 裁判只能给“建议接受/拒绝/继续测试”，不能代替用户裁决
- 如果 B 类判断与 A 类指标冲突，必须解释冲突，不能直接覆盖客观结果

### C 类：必须人工裁决

这类指标涉及用户偏好、业务判断、语气风险、审美和真实场景接受度，LLM 只能整理对比，不能替用户决定。

典型例子：
- 写作语气是否符合用户本人
- 客户、领导、候选人等真实关系中的措辞是否合适
- 设计风格是否符合品牌
- 分析结论是否符合用户的行业经验
- 取舍是否符合用户价值判断

要求：
- Gate 报告必须把 C 类问题明确标注为“需用户裁决”
- 用户不满意时，即使 A 类指标全部通过，也必须继续优化或回退
- 用户满意时，即使 LLM 裁判认为还可改，也不能擅自继续扩改

### 指标分层模板

每个优化任务开始前，先生成以下表格：

| 指标 | 类别 A/B/C | 判断方式 | 阈值或裁决人 | 失败后动作 |
|------|------------|----------|--------------|------------|
| 示例：JSON 可解析 | A | 脚本解析 | 100% 通过 | 优先修复 |
| 示例：语气是否像用户 | C | 用户看左右对比 | 用户裁决 | 继续迭代 |

没有这张表，不得进入 Edit 阶段。

## 第一轮启动协议：v0 → v1 如何选方向

第一轮最容易失控，因为从无 Skill 到 v0，输出变化通常很大；从 v0 到 v1，如果没有严格依据，就会变成“凭感觉改提示词”。

因此第一轮必须按以下步骤选择进化方向：

### 1. 固定任务边界

先确认本轮优化目标只属于以下一种：
- 创建新 Skill：无 Skill → v0 → v1
- 优化已有 Skill：当前 Skill → candidate-v1

如果是创建新 Skill，v0 的目标是“覆盖用户意图和基本流程”，不是追求最优质量。
如果是优化已有 Skill，第一轮不能直接改文件，只能生成候选版本。

### 2. 建立最小真实测试集

第一轮必须至少包含：
- 3 个核心场景：最常见、最能代表 Skill 用途的任务
- 1 个边界场景：容易出错、容易误伤、容易暴露规则冲突的任务
- 1 个回归场景：Skill 原本应该保留的能力

测试用例必须是真实用户会说的话，不能只写抽象需求。

### 3. 跑第一组对照输出

创建新 Skill：
- 执行者跑 without_skill
- 执行者跑 v0
- 裁判对比 v0 是否真正优于 without_skill，并提出 v0 → v1 的修改方向
- 用户先判断裁判方向是否 OK；用户不认可时，不得进入编辑

优化已有 Skill：
- 跑 current_skill
- 如有必要，同时跑 without_skill 或上一已接受版本
- 裁判根据输出判断 current_skill 的问题，并提出 current → candidate-v1 的修改方向
- 用户先判断裁判方向是否 OK；用户不认可时，补测试、重判或结束
- 用户认可方向后，才生成 candidate-v1，再跑 candidate-v1
- 展示 current vs candidate 的完整输出对比，由用户决定是否接受

### 4. 失败分类

Judge 必须先把失败归类，再选择改动方向：
- 交付失败：没有给用户要的结果
- 格式失败：结果不能解析、不能运行、不能打开、结构不合规
- 事实失败：编造、遗漏、错误引用、错误计算
- 流程失败：该问没问、过度追问、跳过关键步骤
- 风格/体验失败：语气、审美、可读性、用户偏好不匹配
- 副作用：新规则破坏旧能力

### 5. 选择第一轮改动方向

第一轮只允许选择 1-3 个高置信方向，按以下优先级：

1. 修复 A 类客观失败
2. 修复“没有交付用户要的结果”这类核心失败
3. 修复多个测试用例共同暴露的同一根因
4. 修复用户明确指出的问题
5. 小幅增强 B/C 类体验问题

禁止第一轮做以下事情：
- 没有真实输出证据就大改结构
- 同时优化太多目标
- 为了一个样例牺牲其他场景
- 把 LLM 裁判建议当成最终结论

### 6. 第一轮必须给用户看什么

第一轮只记住一个核心门槛：每次裁判报告后都停下来，让用户拍板。下面两个位置都遵循同一规则。

**裁判提出方向后，必须展示：**
- 测试用例原文
- without/current 的完整输出
- v0/current 的完整输出
- 裁判提出的 1-3 个修改方向
- 裁判评价表：评价维度、判断方法、证据、置信度、风险、副作用
- 哪些指标是机器判定，哪些是 LLM 建议，哪些需要用户裁决

**裁判复核候选版后，必须展示：**
- 测试用例原文
- current 的完整输出
- candidate 的完整输出
- 逐用例变化和副作用
- 裁判建议：左边好 / 右边好 / 都不够好 / 需要补测试
- 用户裁决区：接受修订并继续 / 接受修订并停止 / 拒绝修订 / 回退 / 补测试

只有用户明确接受，才能把 candidate 应用为新版本。用户叫停时，循环立即结束。

### Phase 2：优化循环（Ratchet Loop）

**目标**：在用户可裁决的监督下迭代优化 Skill。

**每个循环轮次的固定顺序**：

#### 0. Evaluation Contract（评估契约）
- 确认本轮测试用例
- 标注每个指标属于 A 类机器可判、B 类 LLM 辅助判断、C 类人工裁决
- 明确本轮优化目标和不可破坏能力
- 如果指标分层不清，先补评估契约，不进入 Runner

#### 1. Runner：方向基线
- 从训练集（train）中选取 N 个真实用例
- 创建新 Skill 时，跑 without_skill vs v0/current
- 优化已有 Skill 时，跑 current_skill；如有必要，补跑 without_skill 或上一已接受版本
- 保存完整输出、时间、token，以及可自动计算的 A 类指标
- 执行者只保存事实，不写评价，不改 Skill

#### 2. Judge：方向诊断

裁判必须严谨评估，不得只写“感觉更好”。每个判断都要落到维度、方法、证据、置信度和风险。

**诊断数据来源（优先级从高到低）**：
1. **用户反馈**：用户直接指出的问题、不满、偏好、禁区
2. **A 类机器指标**：测试、解析、格式、文件、数值、耗时等明确结果
3. **左右完整输出**：without/current/candidate 的逐例原文
4. **评估契约**：本轮目标、不可破坏能力、A/B/C 分层
5. **历史拒绝记录**：用户曾拒绝过的方向和原因

**裁判评价维度**：

| 维度 | 必查范围 | 判断方法 | 输出要求 |
|------|----------|----------|----------|
| 目标命中 | 所有 Skill | 对照用户原始任务和评估契约 | 说明是否真正交付用户要的东西 |
| 规则遵循 | 所有 Skill | 对照 SKILL.md 的硬规则 | 列出遵循、遗漏、冲突 |
| A 类指标 | 有明确指标的任务 | 脚本、解析器、测试或固定规则 | 给出 pass/fail 或数值 |
| B 类质量 | 结构、逻辑、完整性、明显体验问题 | LLM 辅助判断 | 必须引用输出证据 |
| C 类偏好 | 风格、语气、审美、业务取舍 | 用户裁决 | 只能整理对比，不能替用户决定 |
| 副作用 | 所有修订 | 对照旧能力和回归用例 | 指出新规则可能破坏什么 |
| 泛化风险 | 所有修订 | 看问题是否只来自单一样例 | 标注高/中/低风险 |

**裁判方法要求**：
- 每个建议必须绑定至少一个测试用例和具体输出证据
- 每个建议必须标注来源：A 类机器结果 / B 类辅助判断 / C 类待用户裁决
- 每个建议必须标注置信度：高 / 中 / 低
- 每个建议必须标注可能副作用
- 裁判只能给“建议方向”，不能生成候选 Skill，不能替用户接受或停止

**Judge 方向诊断报告格式**：
```markdown
## Judge 方向诊断报告 v{N}

### 本轮测试范围
| 用例 | 场景 | 左边版本 | 右边版本 | 指标类别 |
|------|------|----------|----------|----------|

### 裁判评价表
| 维度 | 判断方法 | 证据用例 | 结论 | 置信度 | 风险/副作用 | 是否需用户裁决 |
|------|----------|----------|------|--------|------------|----------------|

### 输出对比摘要
- 左边更好的地方：
- 右边更好的地方：
- 都不够好的地方：

### 初步修改方向
1. 方向：
   - 证据：
   - 指标来源：A/B/C
   - 置信度：
   - 可能副作用：
   - 需要用户确认的问题：

### 裁判建议
- 建议：方向 OK / 方向不 OK / 需要补测试
- 理由：
- 必须交给用户裁决的点：
```

#### 3. 用户拍板：是否交给编辑者
- 向用户展示第一组左右完整输出和 Judge 的修改方向
- 用户必须判断：左边好 / 右边好 / 都不够好 / 方向 OK / 方向不 OK / 补测试 / 停止
- 用户可以修正裁判方向；修正后的用户意见优先于裁判原建议
- 用户认可方向后，监督者必须生成“编辑者输入摘要”，再进入 Editor
- 用户不认可方向时，必须补测试、重判、回退或结束；不得继续生成候选版
- 用户叫停、暂停或选择停止时，循环立即截止

**编辑者输入摘要必须包含**：
- 用户最终认可或修正后的方向
- 允许修改的范围
- 禁止修改的范围
- 必须保留的旧能力
- 本轮测试证据链接或摘要
- 需要规避的副作用

#### 4. Editor（候选修订）
- Editor 只能基于“编辑者输入摘要”生成候选版
- 编辑类型包括：补充遗漏规则、修正错误示例、增加或放宽约束、重组章节结构
- Editor 不得自行加入用户未认可的新目标
- **文本学习率预算**：单轮编辑不超过 SKILL.md 总字数的 30%
- **拒绝缓冲区（rejected-edit buffer）**：记录历史上被拒绝的编辑，避免重复
- 候选版只能保存为 `SKILL.candidate.md`，不得覆盖原 Skill

#### 5. Runner：候选复测
- 执行者用相同或约定的测试集跑 current vs candidate
- 保存左右完整输出和 A 类指标
- 执行者仍然不能评价，不能改 Skill

#### 6. Judge：修订复核
- 裁判对 current vs candidate 做同一套评价表
- 裁判必须说明 candidate 是否执行了用户认可的方向
- 裁判必须说明是否出现副作用或新问题
- 裁判只能建议接受、拒绝、部分接受或补测试，不能替用户决定

#### 7. 用户拍板：接受、继续或停止

**本阶段不是自动化判断，而是“结构化呈现 + 用户拍板”。**

**每一轮报告都只是本轮建议，不是最终结论。**

监督者不得把 Judge 的“接受”建议表述为“最终完成”或“最后判断”。正确表述是：
“本轮建议接受/拒绝/继续修改，是否停止由用户决定。”

**候选版本不得直接覆盖原 Skill。**

优化已有 Skill 时，Editor 只能生成候选版本，保存到本轮输出目录，例如：
`outputs/{skill-name}/iterations/iteration-{N}/SKILL.candidate.md`。
在用户裁决前，不得覆盖用户正在使用的原始 `SKILL.md`。只有用户明确接受本轮修改后，
才把候选版本应用到原 Skill；如果用户要求继续优化，则以上一轮候选版本作为下一轮输入，
但仍保持原 Skill 不被覆盖，直到用户接受。用户叫停时，循环立即截止。

**为什么必须用户裁决？**
- LLM 无法判断“语气是否合适”“客户看了会不会满意”等主观体验
- 用户对领域有深度理解，LLM 不具备
- Skill 的优化目标最终是满足用户，不是满足指标

**用户拍板报告输出格式**：
```markdown
## Gate 验证报告 v{N} vs v{N-1}

### 对比概览
| 用例 | 场景 | 指标类别 | 左边关键特征 | 右边关键特征 | 变化方向 |
|------|------|----------|--------------|--------------|----------|

### 逐用例原文对比
（完整输出，不提炼）
- [用例1完整输出左边版本]
- [用例1完整输出右边版本]

### 裁判评价表
| 维度 | 判断方法 | 证据用例 | 结论 | 置信度 | 风险/副作用 | 是否需用户裁决 |
|------|----------|----------|------|--------|------------|----------------|

### 用户意见执行情况
- 用户认可的方向：
- Candidate 是否执行：
- 未执行或偏离的部分：

### 副作用检查
- [ ] 旧场景是否被破坏？
- [ ] 新场景是否被覆盖？
- [ ] 是否有用户未预期的新问题？

### 裁判建议
- 建议：左边好 / 右边好 / 都不够好 / 需要补测试
- 理由：
- 仍需用户裁决的 C 类问题：

### 用户裁决区
- 用户判断：接受修订并继续 / 接受修订并停止 / 拒绝修订 / 回退 / 补测试 / 停止
- 用户修正意见：
```

**决策规则**：
1. **用户认可裁判方向 → 允许生成编辑者输入摘要和候选版**；否则补测试、重判、回退或结束
2. **用户接受修订并继续 → 应用候选为下一轮输入，继续迭代**
3. **用户接受修订并停止 → 应用候选并结束循环**
4. **用户拒绝修订 → 不覆盖原 Skill，记录问题到 rejected_edits.json**
5. **用户不确定 → 补充更多场景测试用例**，再让用户判断
6. **用户叫停、暂停或选择停止 → 循环立即截止**
7. **验证集全部通过但用户不满意 → 以用户为准**，拒绝或继续调整

**关键原则**：
- 每轮 Gate 必须展示**完整原文对比**，不能只有提炼总结
- 用户看到的必须是“左右对照”，而非“我帮你总结的差异”
- 用户拍板是本轮循环的**硬性门槛**
- Gate 报告必须包含：测试用例原始需求、旧版完整输出、候选版完整输出、逐项差异、裁判评价表、裁判建议、用户裁决区
- 如果只做了一轮，不能说“最终版本”；只能说“iteration-{N} 候选版本”
- 用户可以选择接受、继续下一轮、回退、补测试、停止或指定新方向；监督者必须等待或明确邀请用户裁决

**收敛条件**（满足任一即停止）：
- 用户明确接受当前候选并选择停止
- 用户决定暂停、叫停或中止
- 连续 3 轮无 accepted edit，且用户同意停止
- held-out 验证集通过率 ≥ 90%，且用户同意停止
- 达到最大轮次限制（默认 10 轮），进入人工复盘

## LLM 隔离原则

Auto-Skill 涉及多个 LLM 角色，必须严格隔离：

| 角色 | 职责 | 隔离要求 |
|------|------|----------|
| **监督者（Supervisor）** | 主对话中的流程协调者，负责与用户交互和保存记录 | 不直接参与测试、裁判或编辑 |
| **执行者 Runner** | 运行测试用例，收集完整输出和可测指标 | 只产出事实记录，不能评价，不能改 Skill |
| **裁判 Judge** | 根据左右输出提出判断和优化方向 | 只能看输出和评估契约，不能替用户裁决，不能改 Skill |
| **编辑者 Editor** | 根据用户拍板后的编辑者输入摘要生成候选修订 | 不能运行测试，不能扩大用户未认可的修改范围 |
| **复测执行者 Runner** | 运行 current vs candidate 复测 | 只保存左右输出和指标，不能评价 |

**隔离目的**：防止 LLM 自我欺骗——同一个模型既当裁判又当运动员会导致评估偏差。

## Interview 模板（Phase 0）

当用户说"我想做一个 X Skill"时，按以下维度追问：

### 维度 1：范围与分类
- 这个 Skill 处理的任务有哪些类型？（如写作 Skill：公文、汇报、邮件、随笔...）
- 每种类型有什么独特的输入要求或输出格式？
- 用户是否需要先选择类型，还是 Skill 自动判断？

### 维度 2：输出结构与流程
- 任务的完成需要分几步？（如：大纲 → 内容 → 风格校准）
- 每一步的输入输出是什么？
- 是否有必须遵循的固定模板？

### 维度 3：质量与标准
- 输出需要符合什么标准？（如公文 GB/T 9704-2012）
- "好"的输出和"差"的输出区别是什么？
- 有没有需要避免的问题？（如 AI 味过重、格式错误）

### 维度 4：格式与交付
- 支持哪些输出格式？（md / docx / pdf / 其他）
- 是否有特定的排版要求？
- 是否需要引用、附件、元数据？

### 维度 5：示例与反例
- 能否给一个"理想输出"的示例？
- 能否给一个"绝对不能出现"的反例？

## 测试集设计

### 训练集（Train）
- 8-12 个用例
- 覆盖 Skill 的核心场景和边缘情况
- 用于 Rollout 和 Edit 阶段

### Held-out 验证集（Validation）
- 3-5 个用例
- 与训练集不重叠
- 用于 Gate 阶段，验证编辑的泛化能力
- **关键**：验证集只在 Gate 阶段使用，编辑者（Editor）看不到验证集内容

### 测试用例格式
```json
{
  "id": "eval-001",
  "name": "公文-通知",
  "prompt": "帮我写一份关于召开年度工作总结会议的通知",
  "expected_output_summary": "符合 GB/T 9704-2012 的通知格式，包含标题、主送单位、正文、落款",
  "category": "工作类-公文",
  "difficulty": "medium"
}
```

## 编辑策略

### 编辑类型
1. **补充型**：在现有规则后添加新约束（如"增加：公文落款必须包含发文单位全称"）
2. **修正型**：修改错误或过时的规则（如"将'使用宋体'改为'使用仿宋_GB2312'"）
3. **删除型**：移除产生负面效果的规则（如"删除'必须使用首先...其次...最后'的强制结构"）
4. **重组型**：调整章节顺序或合并分散的规则

### 文本学习率预算
- 单轮编辑字数 ≤ SKILL.md 总字数的 30%
- 超过预算时，优先保留高置信度编辑，拆分到低置信度编辑到下一轮

### 拒绝缓冲区
- 维护 `rejected_edits.json`
- 记录每次被拒绝的编辑内容、拒绝理由、对应轮次
- 新编辑提案生成时，先检查是否与缓冲区中的编辑语义重复

## 性能指标

### 必测指标
- **通过率**：输出符合 Skill 定义要求的用例比例
- **时间**：每个用例的完成时间（秒）
- **Token 消耗**：每个用例的总 token 数

### 可选指标（根据 Skill 类型）
- **格式合规率**：输出格式符合规范的比例
- **风格匹配度**：用户风格偏好匹配程度（人工评分）
- **事实准确率**：输出事实正确比例

## 入口：双场景路由

Auto-Skill 必须在任务开始时判断用户意图，走正确的分支：

```
用户输入
    ↓
[检测：用户是否提供了已有 Skill 路径？]
    ↓
    ├── 场景 1：无 Skill（"帮我做一个 X Skill"）
    │       → Phase 0：Interview → 生成 v0 → 用户确认
    │       → Phase 1：基线测试（with_skill vs without_skill）
    │       → Phase 2：优化循环
    │
    └── 场景 2：有 Skill（"帮我优化这个 Skill"）
            → 加载已有 Skill
            → Phase 1：基线测试（with_v{N} vs with_v{N-1}，如无 v{N-1} 则用 without_skill）
            → Phase 2：优化循环（从当前版本开始）
```

**场景 2 的硬性顺序**：
1. 先备份原 Skill，不直接修改原文件
2. 设计真实测试用例，至少覆盖核心场景和一个边界场景
3. 运行旧版输出，保存完整结果
4. 裁判根据旧版输出给出评价表和修改方向，而不是只凭静态阅读下结论
5. 用户看裁判报告后拍板：认可方向、修正方向、补测试、回退或停止
6. 用户认可后，监督者生成编辑者输入摘要；编辑者再生成候选修改，保存为候选文件
7. 运行旧版 vs 候选版输出，保存完整结果
8. 裁判复核候选版是否执行用户意见，并展示完整对比
9. 用户再次拍板：接受并继续、接受并停止、拒绝、补测试、回退或停止

**场景判断信号**：
- 用户说"我想做一个 X Skill"、"帮我创建一个"、"写一个 Skill 出来" → 场景 1
- 用户说"帮我优化 Y Skill"、"这个 Skill 有问题"、"改进一下" + 提供了路径 → 场景 2
- 不确定时询问用户："你是想从零创建一个新 Skill，还是优化已有的 Skill？"

**兼容性要求**（所有生成和执行必须遵循）：
- 所有 Skill 必须兼容通用 LLM 消息格式，不依赖任何厂商或平台专有功能（如特定 thinking 字段、专有 reasoning 字段、专用 IDE/CLI 钩子等）
- 所有脚本使用标准 Python 3.10+，依赖最小化（requests, json, os, subprocess）
- 提示词使用通用 system/user 消息格式，不使用特定供应商专有字段

## 注意事项

1. **用户确认是 Phase 0 的硬性要求**，不可跳过。用户必须看到并确认初版内容。
2. **Held-out 验证集不可泄露**给 Editor，否则失去泛化验证的意义。
3. **每次优化循环都要保留完整数据**（输入、输出、诊断、编辑提案、验证结果），便于复盘。
4. **如果用户中途想停止**，随时可中断，已完成的版本（v0 或中间版本）可直接使用。
5. **优化不是万能的**：如果 Skill 的架构设计有根本缺陷，优化只能改善细节，无法弥补架构问题。此时应回到 Phase 0 重新设计。
6. **不得用静态阅读替代真实案例评估**。静态检查只能用来发现候选问题，不能作为修改依据的唯一证据。
7. **不得把裁判建议冒充用户裁决**。裁判可以建议接受，但最终是否停止、继续或回退，只能由用户决定。
8. **不得在用户裁决前覆盖原 Skill**。所有改动先进入候选文件；用户接受后再应用。
